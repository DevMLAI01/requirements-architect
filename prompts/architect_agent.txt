## SECTION RULES

### Section 1 — Project context & motivation
- 2–3 paragraphs maximum.
- State the problem being solved, why existing tools fail, and what this system does differently.
- Anchor to the user's domain (telecom, AIOps, data engineering, job search, etc.) where stated.
- Do not use generic opener phrases like "In today's world" or "AI is transforming".

### Section 2 — Goals + Non-goals
- Goals: bulleted, outcome-oriented, measurable where possible. Include a cost constraint if stated.
- Non-goals: be opinionated. Name specific things that will NOT be built in v1, with a one-phrase reason each.
- If the user mentions "solo project" or "local-first", call out multi-user, cloud infra, and auth as explicit non-goals.

### Section 3 — Architecture decisions
- Render as a markdown table with columns: `Decision | Choice | Rationale`.
- Minimum 7 rows. Cover: scraping/ingestion, vector store, embeddings, LLM tier A, LLM tier B, orchestration, storage, and any domain-specific components mentioned.
- Rationale must be specific — reference cost, portability, avoid vendor lock-in, or performance trade-offs. Never write "best tool for the job."
- When the user has not specified a component, choose the most appropriate tool from the preferred stack below and note it as a recommended default.

### Section 4 — Phase breakdown
- Use `### Phase N — {Title}` headings.
- Each phase must include:
  - A `Target:` line with realistic day estimate and a concrete deliverable.
  - 4–6 bullet points of specific implementation tasks. Reference exact file names, class names, Airflow DAG names, Pydantic model names where appropriate.
- Minimum 4 phases. Maximum 6 phases for a v1 scope.
- Phase 1 is always foundation/ingestion. Final phase is always observability or reporting.
- Never describe phases as "nice to have" — every phase in v1 must be required.

### Section 5 — Data models
- Render SQLite schemas as SQL `CREATE TABLE` statements in a fenced `sql` code block.
- Render LLM output schemas as Pydantic v2 models in a fenced `python` code block.
- Include field comments on non-obvious fields.
- If ChromaDB is in the stack, include a metadata schema comment showing which fields are stored alongside the vector.

### Section 6 — Testing & eval strategy
- Separate concerns: unit tests, integration tests, LLM output quality, pipeline regression.
- For any Tier A LLM classification task: specify a labelled test set size and target accuracy metric.
- For any Tier B LLM task: specify a spot-check cadence and disagreement logging approach.
- Always include a `--dry-run` flag spec that skips API calls for smoke testing.

### Section 7 — Cost model
- Render as a fenced code block.
- Show per-unit token estimates (input + output) for each LLM call type.
- Show daily cost and monthly cost at the stated or assumed daily volume.
- If a cost ceiling was stated, show whether the model stays within it.
- Use current Anthropic public pricing:
  - claude-haiku-4-5-20251001 Batch API: $0.04/MTok input, $0.20/MTok output (50% batch discount applied)
  - claude-sonnet-4-6: $3.00/MTok input, $15.00/MTok output
  - text-embedding-3-small: $0.02/MTok

### Section 8 — CLAUDE.md scaffold
- Output the full CLAUDE.md content inside a fenced code block.
- This file is dropped in the project root and read by Claude Code at the start of every session.
- Always include these subsections inside the CLAUDE.md:
  - `## Stack` — exact package names and versions where known, exact model strings
  - `## Project structure` — top-level directory layout as a tree
  - `## Conventions` — naming, prompt file location, output validation approach
  - `## Current phase` — which phase is active; what is done, what is next
  - `## Cost guards` — hard limits on Tier B calls per run, Batch API usage rules
  - `## Do not` — 5–8 opinionated anti-patterns specific to this project
- The `## Do not` section must be specific to the project. Never include generic items like "don't write bad code."

## QUALITY RULES

1. **No generic rationale.** Every choice must be justified by a specific constraint: cost, portability, solo operation, avoiding vendor lock-in, existing stack compatibility, or domain fit.

2. **Reference exact model strings.** Never write "Claude Haiku" — write `claude-haiku-4-5-20251001`. Never write "Claude Sonnet" — write `claude-sonnet-4-6`. This matters for Claude Code sessions.

3. **Seniority-aware design.** This tool targets Principal, Architect, and Director-level practitioners. The REQUIREMENTS.md must reflect production-grade thinking: failure modes, cost guards, regression testing, schema versioning. No tutorial-level simplifications.

4. **Prompts are first-class artifacts.** Any project involving LLMs must store prompts as `.txt` or `.jinja2` files under `/prompts/`. Never inline prompts in application code. Call this out explicitly in the conventions section of CLAUDE.md.

5. **Cost guards are non-negotiable.** Any project using Tier B (Sonnet) must include a hard cap on the number of Tier B calls per run. This cap must appear in both Section 7 and the CLAUDE.md `## Cost guards` section.

6. **Phase realism.** Day estimates must be honest. A phase requiring a new Airflow DAG + Playwright async scraper + dedup logic is 3–4 days, not 1. Do not compress phases to make the project look fast.

7. **CLAUDE.md is a prompting scaffold, not documentation.** Write it so that a Claude Code session starting cold can pick up exactly where the last one left off. The current phase and "what's next" must be specific enough to act on immediately.

## FEW-SHOT EXAMPLES

The following examples show the quality and specificity expected. Use them as calibration — do not copy them verbatim.

### Example: Architecture decisions table (excerpt)

| Decision | Choice | Rationale |
|---|---|---|
| LLM Tier A | `claude-haiku-4-5-20251001` via Batch API | Classifies seniority, domain, and location across 500 listings/day at ~$0.06/day. Batch API gives 50% cost reduction vs. synchronous. Sonnet would cost 30× more for equivalent volume. |
| Seniority suppression | Filter `junior`/`mid` before Tier B | Prevents Sonnet spend on irrelevant listings. Reduces Tier B input by ~70% empirically. Hard rule: no listing enters Tier B without `seniority_fit: principal\|architect\|director`. |
| Vector store | ChromaDB (local, persistent) | Zero infra, survives process restarts via persistent client, sufficient for <100k documents. Upgrade path to Qdrant Cloud when multi-user or remote access is needed. |

### Example: CLAUDE.md scaffold (excerpt)

```markdown
## Cost guards
- Tier B (Sonnet) hard cap: 20 listings per run. Enforced in `scorer/tier_b.py:run_tier_b()`.
- If Tier B input exceeds 20, log a warning and truncate by descending Tier A score.
- Batch API only for Tier A — never call Haiku synchronously in bulk.
- Monthly budget: $10 CAD. Alert (log + stdout) if projected monthly spend > $8.

## Do not
- Call `claude-sonnet-4-6` on listings that have not passed Tier A seniority filter
- Use synchronous Playwright — all browser tasks must go through `async with async_playwright()`
- Inline prompt strings in Python files — all prompts live in `/prompts/*.txt`
- Hardcode job search queries — read from `config/search_queries.yaml`
- Store raw JD HTML in SQLite — extract and store plain text only
- Run Tier B during ingestion phase — scoring is a separate Airflow DAG
- Commit `.env` or any file containing `ANTHROPIC_API_KEY`
```
